Creating a Microsoft SQL Server Adaptor for Ecto 2.0

This talk is called Creating a Microsoft SQL Server Adaptor for Ecto 2.0.

It will explore the technology choices made to deliver fast and the developer experience of creating an Ecto adaptor.

Deliver Fast!

  • Only SQL Server features we needed
  • Compose existing technology

We wanted to deliver this project fast, so we made two key decisions on how the project would unfold.

Firstly we would only support the SQL server features we needed, not all of them.

Secondly, we would write as little code as possible, composing already existing tech were we could.

It's such a long way!

Let‚Äôs look at some of the existing tech we used to bridge the gap between Ecto and SQL server.

Currently Microsoft is making moves to be compatible with Linux and mac OS. One of the ways they have done this is with the creation of the Microsoft ODBC Driver for SQL Server.

Getting better...

With that in place we then looked for a pre-existing library that we can use to talk to an ODBC driver, not finding one in Elixir, we looked in Erlang and found Erlang ODBC, part of OTP.

We're half-way there!

With the Microsoft ODBC Driver and Erlang ODBC in place, we have gone from needing to bridge the gap between Ecto and the SQL server binary format, to a smaller gap between Ecto and an Erlang library.

So now begins where we actually write some code.

Divide & Conquer!

We decided to write our implementation as two separate libraries. One would be a general Elixir driver for working with SQL server, encompassing all the ODBC stuff. The other an adaptor between that and Ecto.

MSSQLEX

The general Elixir driver is called MSSQLEX.

DB_CONNECTION

A behaviour module for implementing efficient database connection client processes, pools and transactions.

After looking at similar drivers (Postgres, SQLite) we knew a library called DBConnection is where we should start. This library is a behaviour module, behaviours are quite an underused part of Elixir, but you can think of a behaviour module as similar to an Interface class in OOP.

DBConnection defines a set of behaviours that if implemented through callbacks provides an implementation for database connections, pools and transactions.

We used like a blueprint on how to build our driver in order to get the functionality that we wanted.

Almost there...

After we had implemented the callbacks we had a working Elixir driver for SQL server.

The Project Tomorrow

Erlang ODBC

At the start I said we would only support the SQL server features we needed, next comes fully support all features of SQL server. Unfortunately, the limiting factor is not our libraries, it is instead Erlang ODBC. So we are looking at updating Erlang ODBC to be compatible with the newer features of SQL server and any help from the Elixir / Erlang community would be greatly appreciated.
